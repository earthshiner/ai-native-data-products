Design Review Document

Contract-Aware AI-Native Data Products

A reusable interoperability layer — cross-cutting, not a seventh module.

Teradata defines and runs the product; the Contract Backbone lets open standards consume evidence-backed promises.

Background

The Problem

Three gaps that drove the design.

A contract that lives in a document is not a contract

The existing AI-Native Data Product architecture is deliberately self-describing: each module exposes metadata-rich views, and agents discover capabilities through a structured discovery protocol. This is good design — but it had a gap.

Promises were implicit. The semantic layer described what a product contained, but did not capture what a product promised — the freshness commitment, the null-rate tolerance, the schema stability guarantee. These lived in design documents, if they existed at all.

Validation was optional. Observability tables collected quality evidence, but nothing in the access layer acted on that evidence. A product could be in a known-bad state and still serve data to consumers.

Consumer impact was invisible. When a product changed a contracted interface — adding a column, changing a type, deprecating a field — there was no structured record of which consumers depended on that interface or whether they had acknowledged the change.

The backbone addresses all three: it stores the promise in structured tables, enforces it at the access layer, and records the full history of every decision made about it.

8 gaps against the production data contract standard

A review of the production Data Contract Standard (ODCS) and related article coverage identified eight structural gaps in the prior architecture:

  • No pipeline enforcement gate. Validation results existed but nothing blocked consumption on failure.
  • No rule sub-type enumeration. Quality rules lacked structured categories; agents and export tools could not reason about rule intent.
  • Weak quality coverage floor. There was no minimum rule requirement — a contract interface could exist with zero quality rules.
  • Missing server metadata. The contract had no structured record of where data was served from (host, port, driver).
  • Missing stakeholder registry. No formal owner, data steward, or consumer record was attached to the contract.
  • No tag support. Contracts could not be tagged for domain, sensitivity, or regulatory classification.
  • No SLA formalisation. Freshness, availability, and completeness commitments had no structured representation.
  • No consumer sign-off. Breaking change governance had no structured approval record.
Coverage gaps against 8 open standards

Technical Design

The Design

Five aspects of the contract backbone — explore each tab.

Module Allocation

The backbone does not introduce a new module. It extends three existing module databases — Semantic, Observability, and Memory — with contract-specific tables.

Concern Module Tables
Contract definition Semantic contract, contract_version, contract_interface, contract_field, contract_rule, contract_consumer, contract_server, contract_stakeholder, contract_tag
Validation evidence Observability contract_validation_run, contract_validation_result, contract_violation, contract_sla_status
Governance history Memory contract_change_log, contract_design_decision, contract_consumer_signoff
Contracted data Domain Existing views — not changed
Consumption control Access Publication gate in views; roles tied to interfaces
ℹ
Cross-cutting design standard, not a seventh module. The backbone adds tables to three existing module databases. No new database is created. The access layer enforces the contract through view logic, not a separate gating service.

The Pre-Consumption Enforcement Gate

Previously, validation ran but its outcome had no effect on the access layer. A consumer querying a view would receive data regardless of whether the most recent validation passed or failed. The gate closes that gap.

Publication Status State Machine

UNPUBLISHED
first validation passes
PUBLISHED
CRITICAL failure + BLOCKING
NON_COMPLIANT
violation resolved
PUBLISHED
manual suspend
SUSPENDED
owner lifts
PUBLISHED

BLOCKING — production default

  • CRITICAL rule failure → publication_status = NON_COMPLIANT
  • Access layer view returns no rows
  • Consumers cannot load bad data
  • Requires fix and re-validation to restore

ADVISORY — exploratory use

  • CRITICAL rule failure → alert only
  • Access layer continues serving data
  • Requires documented justification in contract_design_decision
  • Not permitted in production without sign-off

Teradata Gate Check Pattern

The gate is implemented as a correlated EXISTS sub-query in the Access layer view. When publication_status is not PUBLISHED, the sub-query returns no rows, and the view returns no rows.

REPLACE VIEW Customer360_Access_V.customer_current
(
     customer_id
    ,customer_name
    ,customer_status_cd
    ,last_updated_dts
)
AS
LOCKING ROW FOR ACCESS
SELECT
     d.customer_id
    ,d.customer_name
    ,d.customer_status_cd
    ,d.last_updated_dts
FROM Customer360_Domain.customer_current_h AS d
WHERE d.is_current_ind = 1
  AND EXISTS (
    SELECT 1
    FROM Customer360_Semantic.contract_interface AS ci
    WHERE ci.database_name      = 'Customer360_Access_V'
      AND ci.object_name        = 'customer_current'
      AND ci.publication_status = 'PUBLISHED'
      AND ci.is_active          = 1
)
;
ℹ
is_compliant vs publication_status: is_compliant is a convenience boolean alias on the view — 1 when PUBLISHED, 0 for everything else. External tools use is_compliant for a simple yes/no check. Internal pipeline logic uses publication_status to understand why a product is not compliant (non-compliant, suspended, unpublished, or retired).

Integration Coverage

Eight external standards are supported. Direction indicates whether the backbone generates output, receives input, or synchronises both ways.

📋
ODCS
OUTBOUND

contract → schema + quality + SLA + server + stakeholders + tags → ODCS YAML. Full v3 coverage including the fields previously missing (server, stakeholder, tag).

🔌
DPDS
OUTBOUND

contract_interface → all 5 port types: output, input, discovery, observability, and control. Previously only output ports were generated.

📦
ODPS
OUTBOUND

contract + quality evidence → product profile. Includes validation pass rate, active violations, and SLA status in the product descriptor.

🗂
DCAT
OUTBOUND

contract + license_uri + tags → dataset catalogue entry (JSON-LD). Enables registration in CKAN, data.gov, and compatible catalogues.

🔍
DataHub
BIDIRECTIONAL

Assertions + validation results pushed and pulled. Dataset URN pattern:
urn:li:dataset:(urn:li:dataPlatform:teradata,{db}.{table},PROD)

🧪
OpenMetadata
BIDIRECTIONAL

Quality tests + results synchronised. Tests are grouped by contract_interface which maps to a TestSuite. Results flow back into contract_validation_result.

🔗
OpenLineage
OUTBOUND

DataQualityAssertionsFacet + DataQualityMetricsFacet populated on RunEvent. Contract facets added to provide consumer-side context on provenance.

📡
OpenTelemetry
OUTBOUND

5 span attributes + 5 metrics: validation.duration, violations.active, sla.freshness_lag, rules.pass_rate, consumers.registered.

Contract View Groups

Views act as the stable metadata API — agents and tools query views, never base tables. View DDL uses explicit Teradata column lists (column names declared before the AS keyword), so agents can discover the view's columns from the DDL without parsing the SELECT clause.

⚠
All view definitions use explicit Teradata column lists. Agents can discover view columns from the DDL without parsing the SELECT clause — this is a mandatory convention in the design standard.

Discovery Views

*_Semantic_V

Agent and consumer-facing views for contract discovery. The primary entry points for any tool integrating with the backbone.

contract_current contract_current_interface contract_current_field contract_current_rule

Evidence Views

*_Observability_V

Validation history, latest run status, active violations, and SLA standing. Used by monitoring agents and alerting pipelines.

contract_latest_validation contract_validation_history contract_violation_summary contract_sla_status

Consumer Impact Views

*_Observability_V

Registered consumers, impact radius of a violation or breaking change, and deprecated consumer tracking.

contract_registered_consumer contract_consumer_impact contract_deprecated_consumer

Governance Views

*_Memory_V

Full audit trail, design decision log, consumer sign-off status, and breaking change summary for regulatory submission.

contract_change_history contract_signoff_status contract_breaking_change_summary

Standards Export Views

*_Semantic_V

Pre-shaped views for each external standard. Export agents query these views directly — no transformation logic outside the view definition.

contract_odcs_export_v contract_dpds_port_export_v contract_dcat_dataset_export_v contract_lineage_context_v contract_telemetry_context_v

Note: contract_lineage_context_v includes the datahub_dataset_urn column pre-formatted as urn:li:dataset:(urn:li:dataPlatform:teradata,{db}.{table},PROD).

Key Questions for the Team

1

The publication gate returns no rows when NON_COMPLIANT. Is BLOCKING the right default for all production interfaces, or should we default to ADVISORY and let teams opt in to blocking enforcement?

2

The contract floor requires four CRITICAL rules per interface before deployment. Is this too prescriptive? Too lenient? What would you add or remove from the mandatory set?

3

Is the 9-deliverable Add_Contract_To_Data_Product prompt the right granularity for the first implementation? Should we pilot with one or two interfaces before committing to the full 9-deliverable scope?

4

The design adds three new tables to Semantic: contract_server, contract_stakeholder, and contract_tag. Should these be in the first release, or deferred to a follow-up iteration once the core backbone is proven?

5

Are there any external standards or tools from your environment that aren't covered in the integration section? Any proprietary catalogues, lineage tools, or observability platforms we should add?
