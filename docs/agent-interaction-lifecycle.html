1. The Core Principle

An agent that can only read is not an AI-native agent — it's a fast analyst. The separation between ROLE_AGENT and ROLE_READ exists precisely so that agents can write back their interactions, learned strategies, and quality signals without those extended permissions being granted to human consumers.

2. The Lifecycle Flow

A complete agent interaction traverses nine steps across five modules. Steps 1–4 are read-only and require only ROLE_AGENT SELECT permissions. Steps 6–8 are write-back operations that require the INSERT grants exclusive to ROLE_AGENT.

1
Discover the product Semantic READ
{Product}_Semantic
  • Query {Product}_Semantic.data_product_map to find deployed modules and databases.
  • Query {Product}_Semantic.entity_metadata to list all tables and their business purpose.
  • Query {Product}_Semantic.table_relationship to learn join patterns and multi-hop paths.
2
Check contract compliance Semantic Observability READ
{Product}_Semantic_V  ·  {Product}_Observability_V
  • Query {Product}_Semantic_V.contract_current_interface to find contracted interfaces.
  • Check is_compliant = 1 — if 0 (NON_COMPLIANT or SUSPENDED), the Access layer view returns no rows; the agent must not proceed with that interface.
  • Query {Product}_Observability_V.contract_latest_validation for current evidence.
3
Read prior memory Memory READ
{Product}_Memory
  • Query {Product}_Memory.conversation_history for relevant prior interactions with this user or domain.
  • Query {Product}_Memory.learned_strategy for successful query patterns for this entity/question type.
  • Query {Product}_Memory.query_cookbook for pre-built trusted SQL templates.
4
Query the contracted data Domain (via Access_V) READ
{Product}_Access_V
  • Execute SQL against contracted views in {Product}_Access_V — never directly against {Product}_Domain.
  • The Access layer view enforces the publication gate (EXISTS subquery on publication_status = 'PUBLISHED').
  • Uses join patterns from table_relationship and SQL templates from query_cookbook.
5
Process and generate the answer Agent-internal No DB interaction
  • The agent synthesises query results with its language understanding to generate the response.
  • This step is entirely agent-internal — no database interaction occurs.
6
Write conversation record Memory WRITE
{Product}_Memory.conversation_history
  • INSERT into {Product}_Memory.conversation_history — records: session_id, agent_id, user_context, entities_queried, sql_executed, outcome.
  • This is why ROLE_AGENT requires INSERT on Memory; ROLE_READ (humans) does not need this permission.
7
Update learned strategies Memory WRITE (conditional)
{Product}_Memory.learned_strategy  ·  {Product}_Memory.Design_Decision
  • If this interaction produced a novel or successful query pattern, INSERT into {Product}_Memory.learned_strategy.
  • If the agent made a design observation, INSERT into {Product}_Memory.Design_Decision with decision_category = 'AGENT_INSIGHT'.
  • Enables future agents (including itself in later sessions) to improve from this interaction.
8
Record usage event Observability WRITE
{Product}_Observability.telemetry_event
  • INSERT into {Product}_Observability.telemetry_event — records: interface consumed, query cost, row count returned, latency.
  • This is quality signal that feeds back into the contract validation and SLA monitoring loop.
  • Enables the data product team to see how agents are using the product operationally.
9
Return result and close session Agent-internal
  • Session state is persisted in Memory; the next interaction starts at Step 1 but with richer prior context.
  • The Observability write in Step 8 feeds dashboards, contract validation, and OpenTelemetry metrics.

3. The Role Matrix — What Each Role Can Do, and Why

Operation ROLE_READ
(human analysts)		 ROLE_AGENT
(AI agents)		 ROLE_ADMIN
(product owner)
SELECT — Semantic metadata ✓ ✓ ✓
SELECT — Domain (via Access_V views) ✓ ✓ ✓
SELECT — Observability (quality, SLA, lineage) ✓ ✓ ✓
SELECT — Memory (history, strategies, glossary) ✓ ✓ ✓
SELECT — Prediction features ✓ ✓ ✓
INSERT — Memory (conversations, strategies, insights) ✗ ✓ ✓
INSERT — Observability (usage events, feedback) ✗ ✓ ✓
INSERT/UPDATE — Domain tables ✗ ✗ ✓
INSERT/UPDATE — Semantic metadata ✗ ✗ ✓

Why ROLE_AGENT must not write Domain or Semantic: Domain data comes from authoritative source systems via controlled ETL pipelines. Semantic metadata is maintained by data product designers. Agent write-back to these modules would undermine data governance and the integrity of the product's source of truth.

4. Read vs Write Breakdown per Module

Every module has a clear agent access profile. Two modules accept agent writes; four are strictly read-only for agents.

Semantic

Agents READ:

  • Data product map (modules and databases)
  • Entity metadata (tables, business purpose)
  • Table relationships (join patterns, multi-hop paths)
  • Contract interfaces and compliance status
  • Business glossary terms

Agents WRITE:

Nothing — read only. Schema is maintained by data product designers.

{Product}_Semantic
Domain

Agents READ:

  • All contracted data via Access_V views
  • Publication-gated rows only (PUBLISHED status enforced by view)
  • Never directly against base tables

Agents WRITE:

Nothing — read only. Domain data originates from governed ETL pipelines.

{Product}_Domain (via {Product}_Access_V)
Memory

Agents READ:

  • Prior conversation history
  • Learned query strategies
  • Query cookbook templates
  • Design decisions (for context)

Agents WRITE:

  • New conversation records (INSERT)
  • Successful query strategies (INSERT)
  • Agent insights as Design Decisions (INSERT)
{Product}_Memory
Observability

Agents READ:

  • Contract validation results
  • SLA compliance evidence
  • Data lineage records
  • Prior usage telemetry

Agents WRITE:

  • Usage telemetry events (INSERT)
  • Query cost and latency metrics (INSERT)
  • Quality feedback signals (INSERT)
{Product}_Observability
Prediction

Agents READ:

  • Pre-computed prediction features
  • Model output tables
  • Feature definitions and lineage

Agents WRITE:

Nothing — read only. Model training and feature engineering are governed processes.

{Product}_Prediction
Search

Agents READ:

  • Vector embeddings for semantic search
  • Full-text search indexes
  • Ranked retrieval results

Agents WRITE:

Nothing — read only. Embeddings are generated by governed ingestion pipelines.

{Product}_Search

5. Contract and Standards Artifacts

The lifecycle above assumes that standards and contracts are not just narrative documents. They are deployed as queryable artifacts that agents can read, validate, and write evidence against.

A
Standard is authored Memory DESIGN CAPTURE
  • Each standard produces one or more {Product}_Memory.Design_Decision INSERT statements that capture the architectural choice, rationale, and accepted status.
  • The Access Layer produces DD-ACCESS-001; module standards produce their required module design decisions using the same documentation capture protocol.
B
Contract interface is registered Semantic DEPLOYMENT CAPTURE
  • Contracted interfaces are registered in the Semantic module so agents can discover the approved views and expected interface shape.
  • Agents read {Product}_Semantic_V.contract_current_interface before querying business data.
C
Runtime evidence is recorded Observability VALIDATION CAPTURE
  • Validation runs, SLA checks, and usage telemetry are written to Observability as operational evidence.
  • Agents read {Product}_Observability_V.contract_latest_validation to confirm the interface is currently compliant before using it.
In other words: the standards explain the required pattern, the generated SQL INSERTs make that pattern part of product memory, Semantic exposes the current contract interfaces, and Observability records whether those contracts are currently healthy.

6. Implications for the Access Layer DCL

The write-back grants to Memory and Observability are what separates ROLE_AGENT from ROLE_READ at the database level. This is a material permission difference — not just an audit label.

SQL — Access Layer DCL (updated pattern)
-- Phase 1.5 grants — Memory and Semantic
-- SELECT on both (same as ROLE_READ)
GRANT SELECT ON {ProductName}_Semantic  TO {ProductName}_ROLE_AGENT;
GRANT SELECT ON {ProductName}_Memory    TO {ProductName}_ROLE_AGENT;

-- Write-back to Memory: agents record conversations, strategies, and insights
GRANT INSERT ON {ProductName}_Memory    TO {ProductName}_ROLE_AGENT;

-- Phase 2.5 grants — Domain, Observability (and all subsequent modules)
GRANT SELECT ON {ProductName}_Domain        TO {ProductName}_ROLE_AGENT;
GRANT SELECT ON {ProductName}_Observability TO {ProductName}_ROLE_AGENT;

-- Write-back to Observability: agents record usage events and quality feedback
GRANT INSERT ON {ProductName}_Observability TO {ProductName}_ROLE_AGENT;

-- ROLE_READ gets SELECT only — no write-back
GRANT SELECT ON {ProductName}_Semantic      TO {ProductName}_ROLE_READ;
GRANT SELECT ON {ProductName}_Memory        TO {ProductName}_ROLE_READ;
GRANT SELECT ON {ProductName}_Domain        TO {ProductName}_ROLE_READ;
GRANT SELECT ON {ProductName}_Observability TO {ProductName}_ROLE_READ;
The INSERT grants on Memory and Observability are what separates ROLE_AGENT from ROLE_READ at the database level. This is why they are kept as distinct roles — not just for audit clarity, but for materially different database permissions. ROLE_READ remains SELECT-only across all modules.
